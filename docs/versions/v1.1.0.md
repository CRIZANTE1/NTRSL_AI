# v1.1.0 — IA com Gemini (Edge Functions)

**Status:** entregue  
**Escopo:** backend de IA no Supabase; **sem** FastAPI.

## Resumo

Migração do contrato REST planejado em FastAPI para **Supabase Edge Functions** com **Gemini**. A chave `GOOGLE_API_KEY` fica somente nos secrets do projeto Supabase.

## O que entrou

### Cliente

- `src/lib/edgeFunctions.ts` — helper genérico com JWT
- `src/lib/api.ts` refatorado:
  - `postNutritionSummary()` → `nutrition-summary`
  - `postAiRecommendations()` → `ai-recommendations`
  - `getAiCooldown()` → `ai-cooldown`
- **Home:** botão "Calcular resumo" chama Gemini; fallback para `buildSummary()` offline

### Edge Functions (`supabase/functions/`)

| Função | Método | Papel |
|--------|--------|-------|
| `nutrition-summary` | POST | Resumo nutricional via Gemini + tabelas JSON |
| `ai-recommendations` | POST | Coach IA + grava `ai_usage` |
| `ai-cooldown` | GET | Cooldown 30 min por usuário |
| `_shared/` | — | auth, cors, gemini, dados JSON |

### Documentação

- `docs/API.md` reescrito para Edge Functions
- Removidas referências a `VITE_API_BASE_URL` e FastAPI

## Variáveis de ambiente

**App (`.env.local`):**

```env
VITE_SUPABASE_URL=...
VITE_SUPABASE_ANON_KEY=...
```

**Supabase (secrets — não no cliente):**

```env
GOOGLE_API_KEY=<chave-gemini>
GEMINI_MODEL=gemini-2.5-flash   # opcional
```

> Não use `VITE_GEMINI_API_KEY` no app mobile. Guia: [GEMINI_SECRETS.md](../GEMINI_SECRETS.md)

## Deploy (época v1.1.0)

```bash
supabase secrets set GOOGLE_API_KEY=<chave>
supabase functions deploy nutrition-summary
supabase functions deploy ai-recommendations
supabase functions deploy ai-cooldown
```

Ou deploy via MCP Supabase (como no projeto `aumvxnccdhcrftvnliwa`).

## Dependências de banco

Esta versão assume tabela `ai_usage` para cooldown. Sem migration aplicada, cooldown e recomendações podem falhar no servidor.
